# NEXT-001: PLC 角色降级 — 从流程主控到安全守门员

日期：2026-05-03

前置文档：[EVO-002: Motion Stack](../evo/002-motion-stack.md)

状态：草案 / 待项目落地验证

## 背景

传统工站架构中，PLC 同时承担两件事：

1. **流程主控**——决定"什么时候拍照、走哪个点、夹取还是放置"
2. **安全/电气基础设施**——急停回路、门禁互锁、外围执行器断能、24V IO 分发

AutoWeaver 引入 BT + WorldBoard 之后，第 1 件事在编排层级完全可以被吃掉。机械臂控制器（如越疆 Nova 5）自带运动规划、伺服闭环、TCP/IP API，AutoWeaver 直连机械臂在 EVO-002 已经写死为规划路径（`evo/002-motion-stack.md:127`）。

**但 PLC 不能整体被拿走**，因为第 2 件事仍然有真实价值——它涉及硬件失效安全（fail-safe）、安全标准合规（ISO 13849 / IEC 62061）、以及保险与法律责任。

本文档定义"PLC 降级"的边界：哪部分被 AutoWeaver 替代，哪部分必须保留，以及为什么降级后系统仍然安全。

## 重新划分的四层模型

```
┌─────────────────────────────────────────────────────────┐
│ 4. 感知 / 决策层                                         │
│    视觉、检测、坐标修正、策略                              │
├─────────────────────────────────────────────────────────┤
│ 3. AutoWeaver 运动编排层                                 │
│    BT、WorldBoard、Robot TCP Adapter、Rust EtherCAT      │
├─────────────────────────────────────────────────────────┤
│ 2. 设备控制层                                            │
│    机械臂控制盒、伺服驱动器、IO 模块（各自闭环）           │
├─────────────────────────────────────────────────────────┤
│ 1. 安全层                                                │
│    急停、门禁、光栅、安全继电器 / 安全 PLC、              │
│    STO、接触器、气源安全阀                                │
└─────────────────────────────────────────────────────────┘
```

PLC 的位置：从"第 3 层流程主控"**降级**到"第 1 层安全守门员"。

PLC 在降级后**不再做**：

- 下发机械臂运动指令
- 编排"拍照 → 移动 → 夹取 → 放置"节拍
- 处理视觉偏移修正
- 调度多机协同时序
- 维护 HMI 业务画面状态

PLC 在降级后**继续做**（或被等效的安全 PLC / 安全继电器替代）：

- **许可权**：通过 safe_to_move / cell_ready 信号告诉 AutoWeaver"现在允不允许动"
- **否决权**：在不安全状态下强制切断使能、断气源、断危险 DO
- **硬互锁**：急停、门禁、光栅信号的硬接线分发
- **fail-safe 默认态**：上电、断电、复位瞬间各执行器的安全初值

## 为什么要降级

### 1. PLC 作为流程主控是中间商

机械臂控制盒已经原生支持 TCP/IP API，AutoWeaver 完全可以直连。让 PLC 在中间做"翻译 + 转发"是纯粹的冗余层：

```
[降级前]  AutoWeaver → IO 信号 → PLC 寄存器 → 梯形图判断 → TCP → Nova
[降级后]  AutoWeaver → TCP/IP API ────────────────────────────→ Nova
```

每一次转换都是新的 bug 表面积、新的同步语义、新的调试盲区。

### 2. 阻抗不匹配

- AutoWeaver 的业务语义：BT 节点状态 / WorldBoard 键值 / 视觉世界坐标
- PLC 的语义：扫描周期 / 位寄存器 / 梯形图

视觉引导一个偏移 0.3mm 的位姿修正，传统架构下要走"浮点 → IEEE754 拆字 → DB 块 → 梯形图重组 → TCP 报文"的链路，每一步都要做单位与精度对齐。AutoWeaver 直发 TCP 不存在这个问题。

### 3. 维护成本翻倍

改一个工艺意味着同时改两份代码：上位机的 BT/WorldBoard 和 PLC 的梯形图/HMI。调试时要同时看两边的日志，且两边的时间戳通常不同步。

### 4. 性能不仅没收益，反而是负收益

需要把"周期"和"执行耗时"分清楚：

| 指标                    | PLC                          | AutoWeaver BT                  |
|-------------------------|------------------------------|--------------------------------|
| 单次执行耗时            | 整周期扫描全部梯级，毫秒级    | 树遍历活跃路径，亚微秒级        |
| 调度周期                | 扫描周期 5–10 ms（固定）     | tick 间隔 20–50 ms（可调）      |
| 时间复杂度              | O(N)，N = 梯级总数            | O(log N)，N = 节点总数          |
| Nova 反馈周期           | 8 ms（与上位机无关）          | 8 ms（与上位机无关）            |

**BT tick 的实际执行只有亚微秒到微秒级**——一棵 50–200 节点的工业 BT 每次只走活跃分支，dict 查找约 50 ns，整个 tick 在 Python 里就能微秒级完成。tick 间隔是外层 sleep，可以根据业务节拍自由调整，不是性能上限。

加一层 PLC 中转的实际后果：

- **每个指令多走一次 IO 信号 → 寄存器 → 梯形图 → TCP 的转换链**，纯粹增加延迟
- **PLC 扫描周期是固定整周期**，遇到不需要的逻辑也得走完
- **同步语义被强行拆成两段**，AutoWeaver 发完信号还要轮询 PLC 完成位
- **故障排查要同时看两边日志**，时间戳还经常对不上

PLC 既不是瓶颈，也不带来精度收益，引入的纯粹是额外延迟和额外失败模式。

#### 高频反馈场景：以燕窝挑毛为例

上面是单次往返的常规对比，对"间歇下发指令、长动作"的工况差异不太显眼。但**视觉引导的高频修正**场景会把延迟差距放大成节拍差距。

以燕窝挑毛为例：相机检测到一根毛 → 上位机算出偏移 → 下发位姿修正 → 等到位 → 再拍下一帧。这是典型的"每帧都要发指令、每帧都要拿反馈"，闭环频率 10–20 Hz。

**传统链路（AutoWeaver → PLC → Nova）单次往返：**

| 段                              | 典型耗时           |
|---------------------------------|--------------------|
| AutoWeaver → PLC（TCP/Modbus）  | 5–15 ms            |
| PLC 扫描周期等齐边沿              | 5–10 ms（最坏整周期）|
| PLC 梯形图运算                    | 1–3 ms             |
| PLC → Nova（TCP）                | 5–15 ms            |
| Nova 控制盒接收并入队             | 5–10 ms            |
| Nova → PLC 完成回信               | 5–15 ms            |
| PLC → AutoWeaver 完成位轮询       | 5–15 ms（轮询周期决定）|
| **累计**                         | **~30–80 ms，最坏接近 100 ms**|

其中 **PLC 整周期扫描 + AutoWeaver 轮询完成位** 是协议结构性开销，加带宽也救不回来。

**直连链路（AutoWeaver → Nova）单次往返：**

| 段                                     | 典型耗时        |
|----------------------------------------|----------------|
| BT tick 决策                            | 亚微秒级（忽略不计）|
| AutoWeaver → Nova（TCP 29999 / 30003） | 5–15 ms         |
| Nova 控制盒接收入队                       | 5–10 ms         |
| Nova → AutoWeaver（30004 周期反馈）      | 8 ms（订阅式推送）|
| **累计**                                | **~15–30 ms**    |

直连版本是传统方案的 **1/2 到 1/3**。关键收益不是"快一点"，而是结构性消除两段开销：

- BT 决策本身亚微秒级，几乎零成本
- 30004 端口主动推送反馈，AutoWeaver 不用轮询完成位，没有"等 PLC 这一轮扫到我这条梯级"的卡点

**节拍折算：**

- 闭环频率 10–20 Hz × 每次省 30–50 ms = **每秒省 0.3–1 秒等待时间**
- 单工件节拍可能从 3 秒压到 1.5 秒
- 对良率/产能是直接的乘数效应，不是边际优化

这一节也回答了一个常见质疑——"PLC 不是更稳定更实时吗？"。在这种高频反馈场景下，PLC 中转**反而是延迟瓶颈**；它的"稳定"建立在固定扫描周期上，恰恰是在压榨节拍时最碍事的特性。

## 为什么降级后仍然安全

> 核心论点：**安全性来自硬件回路设计与功能安全标准遵循，不来自"是不是 PLC 在编排流程"。**

### 4.1 普通 PLC 本身就不是 Safety-rated

业内常见的认知误区是"有 PLC 就安全"。事实上：

- 普通 S7-1200 / FX5U 等 PLC 不具备 Cat 3 / PL d 等级
- 真正提供安全性的是**独立的安全继电器、安全 PLC、双通道急停按钮、强制导向继电器、接触器反馈检测**
- 这些组件即使在传统架构中也是独立硬接线，从未"依赖普通 PLC 编排"

换句话说，传统架构里 PLC 之所以"显得安全"，是因为它**附近的安全器件**在干活，而不是 PLC 本身。降级方案保留这些器件，只去掉 PLC 这个流程编排者，安全性不下降。

### 4.2 降级方案保留的硬件安全链路

```
急停按钮 NC 双通道
   └─→ 安全继电器（带短路检测、通道一致性检测）
        ├─→ Nova 控制盒 Safety Input
        ├─→ 气源安全阀（断电关闭）
        ├─→ 外部轴使能接触器（强制导向，带反馈）
        └─→ AutoWeaver 旁路读取（仅作 UI 显示与日志）

门禁 / 光栅
   └─→ 安全继电器 / 安全扫描仪
        └─→ Nova Safeguard Input
```

关键设计原则：

- **急停物理回路独立于软件**——AutoWeaver/MCU 永远不能成为停机决策的唯一路径
- **危险外围执行器与急停同步断能**——夹爪气源、真空、外部轴电源都在急停链路里
- **失效即安全**——断线、掉电、信号丢失都触发停机，不需要软件介入

### 4.3 软件层的补强（业务级保护，不是替代硬件安全）

软件这一层做的是 **operational safety**（运行级保护），不是 safety-rated circuit：

- **BT 内的 SafetyCondition**：门关、真空建立、路径区清空、机器人 idle、无报警等软件互锁
- **三级 watchdog**：AutoWeaver Python ↔ Rust runtime ↔ MCU 互看心跳
- **fail-safe 默认值**：TCP 断连 / 反馈超时 / 运动超时统一走 Stop + 关 DO
- **进程隔离**：Python 崩溃时 Rust runtime 主动 disable axis；Rust 崩溃时硬件回路独立工作

### 4.4 软件停机 vs 硬件停机

| 维度       | 软件停机（BT halt / TCP Stop） | 硬件停机（E-stop / Safeguard） |
|------------|-------------------------------|-------------------------------|
| 触发延迟   | 一个 tick（20–50 ms）          | 继电器响应时间（< 20 ms）       |
| 失效模式   | 软件 bug / TCP 阻塞 / GIL 卡死 | 失效即安全（断线即停）          |
| 决策依据   | BT 条件 / 业务逻辑             | 电气状态 / 物理输入             |
| 适用场景   | 业务异常、感知失败、超时         | 急停、门禁、光栅、过流          |

两者并行存在，**互为冗余**而不是互相替代。AutoWeaver 拿走的是流程编排，不是硬件停机回路。

- [ ] `kill -9` AutoWeaver 主进程 → 机械臂当前轨迹完成后停止，且不接受新指令

- [ ] Rust runtime 崩溃 → EtherCAT IO 输出回到 fail-safe 默认值

- [ ] 完成 ISO 13849 风险评估文档（外部功能安全顾问签字）
- [ ] 急停回路达到 PL d Cat 3 或更高
- [ ] 明确停止类别选择记录（默认 Cat 1，特殊场景 Cat 0）
- [ ] 留存第三方安全验证测试报告

### 工程验证

- [ ] 24V 电气柜 EMC 测试（MCU 接管普通 IO 后特别需要）
- [ ] 双周连续运行测试（统计 false stop 次数）
- [ ] 上电 / 断电 / 复位三种瞬态下 fail-safe 默认态成立
- [ ] 远程 OTA 升级时机器人处于安全态（不能在运动中升级）

## 边界与已知风险

### 明确不要做的事

1. **不要**把急停按钮接到 MCU GPIO 后由 MCU 软件判断是否发 Stop
2. **不要**让 AutoWeaver 软件互锁成为唯一停机路径
3. **不要**在没有功能安全顾问参与的情况下推到量产
4. **不要**简化掉"看起来多余"的强制导向继电器、接触器反馈这类元件
5. **不要**因为"协作机器人有内置安全"就跳过外部急停链路设计

### 已知盲区（待 EVO-006 / 后续 spec 解决）

- AutoWeaver 目前没有形式化的 Safety Monitor 模块，建议立项 **EVO-006: Safety Monitor**
- watchdog 心跳协议在 `evo/003-motion-runtime.md` 只是"提到要做"，没有规约
- 三级 watchdog（Python ↔ Rust ↔ MCU）握手语义需要单独 spec
- 视觉引导失败时的 fallback 策略尚无统一约定（continue / abort / human review）
- 多机协同场景下"谁的急停影响谁"的范围划分未定义

### 文档口径需要修订（已识别但未提交）

- `evo/001-motion-engine.md:24` 当前写"e-stop must halt motion within one control cycle"——容易被误读为软件 tick 即可保证急停。建议改为"软件保护停机在一个 tick 内发出 Stop；硬件 E-stop 由独立安全回路触发"。
- `evo/003-motion-runtime.md` 中"Python 崩溃不影响电机安全"——需要补充 watchdog 条款：心跳丢失时 Rust 主动 quick stop / disable / 关危险 DO，不能仅依赖"保持当前状态"。

## 商业化与组织阻力

PLC 角色降级的技术路线清晰，但落地时存在**非技术阻力**：

- 现场 PLC 工程师是工厂内方案评审的实际把关人
- AutoWeaver 方案明摆着冲击他们的核心岗位价值
- 中位数 PLC 工程师月薪 8k–15k，会做 ISO 13849 / 安全签字的不到 5%
- 被替代的多数人在岗位市场上可能落到"高级调试技工"档位（6–9k）

推进策略上的取舍：

| 路径                     | 优点                       | 风险                         |
|--------------------------|---------------------------|------------------------------|
| 直接打老板，绕过 PLC 团队  | 决策最快                   | 实施期 PLC 团队消极配合       |
| 包装成"产线编排工程师"工具 | PLC 团队不主动反对         | 转型门槛筛掉一部分人          |
| 选愿意转型的工站做 pilot   | 阻力最小，案例可复制       | 推进慢                        |

这一节不属于技术 spec，但项目落地阶段必须提前规划。
